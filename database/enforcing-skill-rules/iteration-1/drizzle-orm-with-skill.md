# Drizzle ORM — With Skill

## Prompt
Set up Drizzle ORM schema for users table with id, name, email, role, timestamps. Show schema definition, type generation, primary key setup, constraints/indexes. Should CLI-generated output be used as-is?

## Response

### 1. Drizzle Schema Definition

```typescript
import { pgTable, text, timestamp, uuid, check, uniqueIndex, index } from 'drizzle-orm/pg-core';
import { sql, InferSelectModel } from 'drizzle-orm';
import { uuidv7 } from 'uuidv7';

export const users = pgTable(
  'users',
  {
    id: uuid('id')
      .primaryKey()
      .$defaultFn(() => uuidv7()),  // UUIDv7: time-sortable, index-friendly
    name: text('name').notNull(),
    email: text('email').notNull(),
    role: text('role').notNull().default('member'),
    created_at: timestamp('created_at', { withTimezone: true })  // Always withTimezone: true
      .notNull()
      .defaultNow(),
    updated_at: timestamp('updated_at', { withTimezone: true })
      .notNull()
      .defaultNow(),
  },
  (table) => [
    // Constraints and indexes in the third argument
    uniqueIndex('users_email_idx').on(table.email),
    index('users_role_idx').on(table.role),
    check('users_role_check', sql`${table.role} IN ('admin', 'member', 'viewer')`),
  ]
);
```

### 2. Type Generation — InferSelectModel

Export the row type directly from the schema file — **schema = source of truth**:

```typescript
// Same file as the schema definition
export type User = InferSelectModel<typeof users>;
// User = { id: string; name: string; email: string; role: string; created_at: Date; updated_at: Date }
```

### 3. Primary Key — UUIDv7

The `.$defaultFn(() => uuidv7())` generates UUIDv7 at the application layer:
- **Time-sortable**: contains a timestamp prefix, so B-tree inserts are sequential (good index locality)
- **Avoids UUIDv4**: random UUIDs cause index page splits and poor cache locality on clustered keys

### 4. Relations (for query builder)

```typescript
import { relations } from 'drizzle-orm';

export const usersRelations = relations(users, ({ many }) => ({
  projects: many(projects),
}));
```

### 5. CLI Output — Starting Point, Not Production-Ready

**Do not use `drizzle-kit generate` output as-is in production.** The CLI-generated schema is a starting point that typically needs manual adjustment for:

- `{ withTimezone: true }` — CLI may generate bare `timestamp()` without timezone
- CHECK constraints — not generated by CLI
- Custom indexes — CLI generates basic ones, not partial/covering indexes
- snake_case columns — CLI may not respect your naming conventions
- UUIDv7 `$defaultFn` — CLI uses `defaultRandom()` which gives UUIDv4

Always review and adjust CLI output against your schema standards before committing.
