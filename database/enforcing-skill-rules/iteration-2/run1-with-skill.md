## 1. SQL Schema Issues

### Issue 1 — TRUNCATE under traffic (Gotcha: TRUNCATE lock)

**Rule**: *`TRUNCATE` takes ACCESS EXCLUSIVE lock, not MVCC-safe. Use `DELETE FROM` during traffic.*

```sql
-- BAD
TRUNCATE TABLE audit_logs;

-- FIXED
DELETE FROM audit_logs WHERE created_at < NOW() - INTERVAL '90 days';
-- Or if full wipe is needed during maintenance window:
-- TRUNCATE TABLE audit_logs;  -- only during zero-traffic window
```

**Why**: `TRUNCATE` acquires an `ACCESS EXCLUSIVE` lock, blocking all concurrent reads and writes. On a live system this causes outages. `DELETE FROM` uses MVCC and is safe under traffic. Batch-delete old rows with a WHERE clause for even less lock contention.

---

### Issue 2 — TIMESTAMP without time zone (Critical Rule: Schema Design)

**Rule**: *Types: TIMESTAMPTZ not TIMESTAMP.*

```sql
-- BAD (all tables)
created_at TIMESTAMP DEFAULT NOW()

-- FIXED
created_at TIMESTAMPTZ DEFAULT NOW()
```

**Why**: Bare `TIMESTAMP` stores no timezone info. If the server, client, or session timezone changes, times silently shift. `TIMESTAMPTZ` stores UTC internally and converts on display — always correct.

---

### Issue 3 — VARCHAR instead of TEXT (Critical Rule: Schema Design)

**Rule**: *Types: TEXT not VARCHAR.*

```sql
-- BAD
name VARCHAR(100) NOT NULL,
full_name VARCHAR(255) NOT NULL,
email VARCHAR(255) UNIQUE,

-- FIXED
name TEXT NOT NULL,
full_name TEXT NOT NULL,
email TEXT UNIQUE NOT NULL,
```

**Why**: In PostgreSQL, `VARCHAR(N)` and `TEXT` have identical performance. `VARCHAR(N)` just adds an arbitrary length constraint that causes migration pain when business needs change. Use `TEXT` with `CHECK` constraints if length limits are needed.

---

### Issue 4 — UUIDv4 instead of UUIDv7 for PKs (Critical Rule: Schema Design)

**Rule**: *PKs: UUIDv7 (index locality, time-sortable). Never random UUIDv4 for clustered keys.*

```sql
-- BAD
id UUID DEFAULT gen_random_uuid() PRIMARY KEY,

-- FIXED — requires uuid_v7 extension or app-generated UUIDv7
id UUID PRIMARY KEY,  -- UUIDv7 generated by application layer
```

**Why**: `gen_random_uuid()` produces UUIDv4 which is fully random, causing B-tree index page splits and write amplification. UUIDv7 is time-sorted, so inserts are append-only — dramatically better write performance at scale.

---

### Issue 5 — FLOAT for money (Critical Rule: Schema Design)

**Rule**: *NUMERIC for money never FLOAT.*

```sql
-- BAD
budget FLOAT,
price_estimate FLOAT,

-- FIXED
budget NUMERIC(12, 2) NOT NULL CHECK (budget > 0),
price_estimate NUMERIC(10, 2) NOT NULL CHECK (price_estimate > 0),
```

**Why**: IEEE 754 floating point cannot represent `0.1` exactly. Financial calculations accumulate rounding errors. `NUMERIC` stores exact decimal values.

---

### Issue 6 — Soft delete with boolean (Critical Rule: Schema Design)

**Rule**: *Soft delete: `deleted_at TIMESTAMPTZ` with partial unique index. Never boolean `is_deleted`.*

```sql
-- BAD
is_deleted BOOLEAN DEFAULT false,

-- FIXED
deleted_at TIMESTAMPTZ DEFAULT NULL,
```

**Why**: A boolean loses *when* the deletion happened, making audit trails and time-based recovery impossible. `deleted_at TIMESTAMPTZ` gives you both the "is deleted?" check (`WHERE deleted_at IS NULL`) and the timestamp.

---

### Issue 7 — JSON stored as TEXT (Critical Rule: Schema Design)

**Rule**: *JSONB for semi-structured.*

```sql
-- BAD
preferences TEXT, -- JSON stored as text

-- FIXED
preferences JSONB DEFAULT '{}',
```

**Why**: `TEXT` has no JSON validation — invalid JSON is silently stored. `JSONB` validates on insert, supports indexing (`GIN`), and enables queries like `preferences->>'theme'`.

---

### Issue 8 — API key stored in plain text (Critical Rule: Security)

**Rule**: *bcrypt/argon2 for passwords. Never store API keys/secrets plain text.*

```sql
-- BAD
api_key TEXT, -- stored in plain text

-- FIXED
api_key_hash TEXT, -- store argon2/bcrypt hash; show key once at creation
```

**Why**: If the database is compromised, all API keys are immediately usable. Hash with argon2id. Display the raw key only once at creation time. For lookup, store a prefix or use a hash-based lookup index.

---

### Issue 9 — No CHECK constraints (Critical Rule: Schema Design)

**Rule**: *CHECK constraints: `price > 0`, `status IN (...)`. Make invalid data unrepresentable.*

```sql
-- BAD
status TEXT,

-- FIXED
status TEXT NOT NULL DEFAULT 'backlog'
  CHECK (status IN ('backlog', 'active', 'in_progress', 'done', 'cancelled')),
```

**Why**: Without constraints, any string can be inserted into status columns. This pushes validation entirely to the app layer, and bugs or direct DB access can corrupt data.

---

### Issue 10 — Missing FK indexes (Gotcha + Critical Rule: Indexing)

**Rule**: *FK columns need explicit indexes — NOT auto-indexed.* and *Index ALL FK columns — prevents cascade lock issues.*

```sql
-- MISSING — add these
CREATE INDEX idx_users_org_id ON users(org_id);
CREATE INDEX idx_projects_org_id ON projects(org_id);
CREATE INDEX idx_tasks_project_id ON tasks(project_id);
CREATE INDEX idx_tasks_assignee_id ON tasks(assignee_id);
```

**Why**: PostgreSQL does NOT auto-index FK columns (unlike MySQL). Without indexes, every `DELETE` on the parent table triggers a sequential scan on the child table while holding a lock.

---

```sql
-- With soft delete, unique email should exclude deleted rows
```

## 2. Query Issues

### Issue 12 — SELECT * (Critical Rule: Performance)

**Rule**: *No SELECT *: explicit columns enables covering indexes.*

```sql
-- BAD
SELECT * FROM tasks WHERE project_id = $1 AND status = 'active' ORDER BY created_at DESC;

-- FIXED
SELECT id, project_id, title, status, price_estimate, created_at
FROM tasks
WHERE project_id = $1 AND status = 'active'
ORDER BY created_at DESC;
```

**Why**: `SELECT *` fetches all columns including large TEXT fields, wastes bandwidth, defeats covering indexes, and breaks when columns are added/removed.

---

### Issue 13 — LIKE '%term%' for text search (Critical Rule: Performance)

**Rule**: *Text search: TSVECTOR + GIN. Never `LIKE '%term%'` on large tables.*

```sql
-- BAD
SELECT * FROM tasks WHERE description LIKE '%deploy%';

-- FIXED — add tsvector column + GIN index
ALTER TABLE tasks ADD COLUMN description_tsv TSVECTOR
  GENERATED ALWAYS AS (to_tsvector('english', COALESCE(description, ''))) STORED;
CREATE INDEX idx_tasks_description_tsv ON tasks USING GIN(description_tsv);

-- Query
SELECT id, title, description, status
FROM tasks
WHERE description_tsv @@ plainto_tsquery('english', 'deploy');
```

**Why**: `LIKE '%term%'` forces a full sequential scan — it cannot use any B-tree index. `TSVECTOR + GIN` provides O(log N) full-text search with ranking.

---

### Issue 14 — OFFSET pagination on deep pages (Critical Rule: Performance)

**Rule**: *Pagination: cursor-based (keyset). OFFSET/LIMIT is O(N) on deep pages.*

```sql
-- BAD
SELECT * FROM tasks WHERE assignee_id = $1 ORDER BY created_at DESC LIMIT 20 OFFSET 200;

-- FIXED — cursor-based (keyset) pagination
SELECT id, title, status, created_at
FROM tasks
WHERE assignee_id = $1
  AND (created_at, id) < ($2, $3)  -- cursor from previous page
ORDER BY created_at DESC, id DESC
LIMIT 20;
```

**Why**: `OFFSET 200` reads and discards 200 rows before returning 20. At `OFFSET 10000` the DB reads 10020 rows. Keyset pagination uses an index seek — constant time regardless of page depth.

---

### Issue 15 — Missing composite/partial index for active tasks query

**Rule**: *Index based on query patterns (WHERE, JOIN, ORDER BY).*

```sql
-- Support the active tasks query pattern
CREATE INDEX idx_tasks_project_status_created
  ON tasks(project_id, status, created_at DESC)
  WHERE status = 'active';
```

**Why**: The most frequent query filters by `project_id + status = 'active'` and sorts by `created_at DESC`. A partial composite index serves this exactly with minimal disk/RAM cost.

---

### Issue 16 — NOW() gotcha not addressed (Gotcha)

**Rule**: *`NOW()` in transaction = same value entire txn. Use `clock_timestamp()` for wall-clock.*

```sql
-- BAD (inside a long transaction)
SELECT * FROM tasks WHERE created_at > NOW() - INTERVAL '1 hour';

-- FIXED — if this runs inside a transaction and needs wall-clock time:
SELECT id, title, status, created_at
FROM tasks
WHERE created_at > clock_timestamp() - INTERVAL '1 hour';
```

## 3. Connection Pool Issue

### Issue 17 — Pool exhaustion at max_connections (Gotcha)

**Rule**: *Pool math: 5 instances x 20 pool = 100 = default max_connections limit. Use PgBouncer.*

5 instances x 20 connections = 100 = PostgreSQL's default `max_connections`. Zero headroom for admin connections, monitoring, migrations, or connection spikes.

**Fix**: Add PgBouncer in transaction mode. App connects to PgBouncer (pool of ~20 per instance), PgBouncer maintains ~50 actual DB connections.

```
; pgbouncer.ini
[databases]
myapp = host=localhost dbname=myapp

[pgbouncer]
pool_mode = transaction
default_pool_size = 50
max_client_conn = 200
```

Reduce each app instance pool to 10. PgBouncer multiplexes to ~50 server connections, leaving headroom.

## 4. Migration Script Issues

### Issue 18 — Rename column without expand-and-contract (Critical Rule: Migrations)

**Rule**: *Expand-and-Contract for breaking changes.*

```sql
-- BAD — breaks all running queries instantly
ALTER TABLE users RENAME COLUMN name TO full_name;

-- FIXED — expand and contract
-- Step 1: Add new column
ALTER TABLE users ADD COLUMN full_name TEXT;
-- Step 2: Backfill
UPDATE users SET full_name = name WHERE full_name IS NULL;
-- Step 3: App reads from COALESCE(full_name, name), writes to both
-- Step 4: Deploy app reading only full_name
-- Step 5: Drop old column
ALTER TABLE users DROP COLUMN name;
```

---

### Issue 19 — SET NOT NULL without NOT VALID (Critical Rule: Migrations)

**Rule**: *ADD CONSTRAINT NOT VALID then VALIDATE separately — avoid table locks.*

```sql
-- BAD — full table scan + ACCESS EXCLUSIVE lock
ALTER TABLE users ALTER COLUMN email SET NOT NULL;

-- FIXED
ALTER TABLE users ADD CONSTRAINT users_email_not_null
  CHECK (email IS NOT NULL) NOT VALID;
-- Then in a separate migration/transaction:
ALTER TABLE users VALIDATE CONSTRAINT users_email_not_null;
```

**Why**: `SET NOT NULL` scans the entire table under an exclusive lock. The two-step approach lets the validation happen without blocking writes.

---

### Issue 20 — CREATE INDEX without CONCURRENTLY (Critical Rule: Migrations)

**Rule**: *CREATE INDEX CONCURRENTLY.*

```sql
-- BAD — blocks all writes on tasks table
CREATE INDEX idx_tasks_project ON tasks(project_id);

-- FIXED
CREATE INDEX CONCURRENTLY idx_tasks_project ON tasks(project_id);
```

**Why**: Non-concurrent index creation takes a `SHARE` lock blocking all `INSERT/UPDATE/DELETE` for the duration. On a large table this means downtime.

---

### Issue 21 — Missing lock_timeout (Critical Rule: Migrations)

**Rule**: *`SET lock_timeout = '5s'` on every DDL migration.*

```sql
-- Add to top of every migration
SET lock_timeout = '5s';
```

**Why**: Without `lock_timeout`, DDL statements can queue behind long-running transactions indefinitely, cascading into a full write outage.

---

### Issue 22 — Disabling autovacuum (Critical Rule: Migrations)

**Rule**: *Tune autovacuum frequency on high-churn tables. NEVER disable autovacuum.*

```sql
-- BAD
ALTER TABLE tasks SET (autovacuum_enabled = false);

-- FIXED — tune it instead
ALTER TABLE tasks SET (
  autovacuum_vacuum_scale_factor = 0.01,
  autovacuum_analyze_scale_factor = 0.005,
  autovacuum_vacuum_cost_delay = 2
);
```

**Why**: Disabling autovacuum causes dead tuple bloat, index bloat, and eventually transaction ID wraparound — which forces PostgreSQL into single-user emergency mode. Tune the parameters instead.

## 5. Drizzle Schema Issues

### Issue 23 — Missing withTimezone, wrong types, no constraints, no type export (Critical Rule: Drizzle ORM)

**Rule**: *Timestamps: always `{ withTimezone: true }`. UUIDv7. Export `InferSelectModel`. CHECK constraints.*

```typescript
// schema.ts — FIXED
import {
  pgTable, uuid, text, timestamp, numeric, serial, check, index,
} from 'drizzle-orm/pg-core';
import { sql } from 'drizzle-orm';
import type { InferSelectModel } from 'drizzle-orm';
import { uuidv7 } from 'uuidv7'; // app-generated UUIDv7

export const tasks = pgTable(
  'tasks',
  {
    id: serial('id').primaryKey(),
    projectId: uuid('project_id').references(() => projects.id).notNull(),
    assigneeId: uuid('assignee_id').references(() => users.id),
    title: text('title').notNull(),
    description: text('description'),
    priceEstimate: numeric('price_estimate', { precision: 10, scale: 2 }).notNull(),
    status: text('status').notNull().default('backlog'),
    createdAt: timestamp('created_at', { withTimezone: true }).defaultNow().notNull(),
  },
  (table) => [
    check('tasks_price_positive', sql`${table.priceEstimate} > 0`),
    check('tasks_status_valid', sql`${table.status} IN ('backlog', 'active', 'in_progress', 'done', 'cancelled')`),
    index('idx_tasks_project_id').on(table.projectId),
    index('idx_tasks_assignee_id').on(table.assigneeId),
  ],
);

// Always export the inferred type alongside the table
export type Task = InferSelectModel<typeof tasks>;
```

**Sub-issues fixed**:
- `timestamp()` -> `timestamp({ withTimezone: true })`
- `real()` -> `numeric()` (no FLOAT for money)
- Missing `assigneeId`, `description` columns added
- Added CHECK constraints in third argument
- Added FK indexes in third argument
- Exported `Task` type via `InferSelectModel`
- Added `.notNull()` where appropriate
- Noted that drizzle-kit output is a starting point, not production-ready

## 6. Repository Layer Issues

### Issue 24 — SQL injection via string concatenation (Critical Rule: Security)

**Rule**: *Parameterized queries only. Never concatenate strings into SQL.*

```typescript
// BAD — SQL injection
async search(userInput: string) {
  return await db.execute(`SELECT * FROM tasks WHERE title LIKE '%${userInput}%'`);
}

// FIXED — use Drizzle's built-in operators
import { like } from 'drizzle-orm';

async search(userInput: string): Promise<Task[] | DatabaseError> {
  try {
    return await db
      .select({
        id: tasks.id,
        title: tasks.title,
        status: tasks.status,
        createdAt: tasks.createdAt,
      })
      .from(tasks)
      .where(like(tasks.title, `%${userInput}%`));
  } catch (err) {
    return new DatabaseError('search failed', { cause: err });
  }
}
```

**Why**: Direct string interpolation allows an attacker to inject arbitrary SQL (`'; DROP TABLE tasks; --`). Drizzle's query builder uses parameterized queries internally.

### Issue 25 — Throwing errors instead of returning error values (Critical Rule: Repository Pattern)

**Rule**: *Return errors as values, never throw. Domain-specific error classes.*

```typescript
// errors.ts
export class TaskNotFoundError {
  readonly _tag = 'TaskNotFoundError' as const;
  constructor(readonly id: string) {}
}

export class DatabaseError {
  readonly _tag = 'DatabaseError' as const;
  constructor(readonly message: string, readonly options?: { cause?: unknown }) {}
}

// task-repository.ts — FIXED
import { db } from './db';
import { tasks, type Task } from './schema';
import { eq } from 'drizzle-orm';
import { TaskNotFoundError, DatabaseError } from './errors';

export function createTaskRepository() {
  return {
    async findById(id: string): Promise<Task | TaskNotFoundError | DatabaseError> {
      try {
        const rows = await db
          .select()
          .from(tasks)
          .where(eq(tasks.id, id));

        if (!rows.length) return new TaskNotFoundError(id);
        return rows[0];
      } catch (err) {
        return new DatabaseError('findById failed', { cause: err });
      }
    },

    async createWithComment(
      taskData: typeof tasks.$inferInsert,
      commentText: string,
    ): Promise<{ task: Task; comment: Comment } | DatabaseError> {
      try {
        return await db.transaction(async (tx) => {
          const [task] = await tx.insert(tasks).values(taskData).returning();
          const [comment] = await tx.insert(comments).values({
            taskId: task.id,
            text: commentText,
          }).returning();
          return { task, comment };
        });
      } catch (err) {
        return new DatabaseError('createWithComment failed', { cause: err });
      }
    },
  };
}

// Derive type from factory
export type TaskRepository = ReturnType<typeof createTaskRepository>;
```

**Sub-issues fixed**:
- `throw new Error` replaced with returned error values
- Domain-specific `TaskNotFoundError` class
- `DatabaseError` wrapping unexpected failures with `cause`
- Factory function pattern with derived type
- Typed `taskData` parameter (no `any`)
- Transaction error union documented in return type

## 7. Database Client & Access Control Issues

### Issue 27 — Superuser access + no RLS + no OTel logger (Critical Rules: Security + OTel)

**Rule**: *App never superuser. GRANT only SELECT/INSERT/UPDATE/DELETE.* and *RLS on every table.* and *OtelDrizzleLogger.*

```typescript
// db.ts — FIXED
import { drizzle } from 'drizzle-orm/node-postgres';
import { Pool } from 'pg';
import { trace, Span } from '@opentelemetry/api';
import * as schema from './schema';

// OTel query logger (zero overhead when no active span)
class OtelDrizzleLogger {
  logQuery(query: string, params: unknown[]): void {
    const span = trace.getActiveSpan();
    if (!span) return;
    span.setAttribute('db.system', 'postgresql');
    span.setAttribute('db.statement', query);
  }
}

// Use a non-superuser role
const pool = new Pool({
  connectionString: process.env.DATABASE_URL, // must point to app_user, not postgres
  max: 10, // reduced from 20 — PgBouncer handles multiplexing
});

export const db = drizzle(pool, {
  schema,
  logger: new OtelDrizzleLogger(),
});
```

```sql
-- Create least-privilege app role
CREATE ROLE app_user LOGIN PASSWORD '...';
GRANT CONNECT ON DATABASE myapp TO app_user;
GRANT USAGE ON SCHEMA public TO app_user;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO app_user;
GRANT USAGE ON ALL SEQUENCES IN SCHEMA public TO app_user;

-- Enable RLS on every table
ALTER TABLE organizations ENABLE ROW LEVEL SECURITY;
ALTER TABLE users ENABLE ROW LEVEL SECURITY;
ALTER TABLE projects ENABLE ROW LEVEL SECURITY;
ALTER TABLE tasks ENABLE ROW LEVEL SECURITY;

-- Example policy: users see only their org's tasks

  USING (project_id IN (
    SELECT id FROM projects WHERE org_id = current_setting('app.current_org_id')::uuid
  ));
```

## Complete Fixed Schema

```sql

CREATE EXTENSION IF NOT EXISTS pgcrypto;

-- Lock timeout for all DDL
SET lock_timeout = '5s';

CREATE TABLE organizations (
  id UUID PRIMARY KEY,              -- UUIDv7 from app
  name TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE users (
  id UUID PRIMARY KEY,              -- UUIDv7 from app
  org_id UUID NOT NULL REFERENCES organizations(id),
  full_name TEXT NOT NULL,
  email TEXT NOT NULL,
  password_hash TEXT NOT NULL,       -- argon2id
  api_key_hash TEXT,                 -- argon2id hash, never plain text
  deleted_at TIMESTAMPTZ DEFAULT NULL,
  preferences JSONB DEFAULT '{}',
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE UNIQUE INDEX idx_users_email_active ON users(email) WHERE deleted_at IS NULL;
CREATE INDEX idx_users_org_id ON users(org_id);

CREATE TABLE projects (
  id UUID PRIMARY KEY,              -- UUIDv7 from app
  org_id UUID NOT NULL REFERENCES organizations(id),
  name TEXT NOT NULL,
  budget NUMERIC(12, 2) NOT NULL CHECK (budget > 0),
  status TEXT NOT NULL DEFAULT 'active'
    CHECK (status IN ('active', 'archived', 'cancelled')),
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_projects_org_id ON projects(org_id);

CREATE TABLE tasks (
  id SERIAL PRIMARY KEY,
  project_id UUID NOT NULL REFERENCES projects(id),
  assignee_id UUID REFERENCES users(id),

    to_tsvector('english', COALESCE(description, ''))
  ) STORED,

);

-- Tune autovacuum on high-churn table (NEVER disable)
ALTER TABLE tasks SET (
  autovacuum_vacuum_scale_factor = 0.01,
  autovacuum_analyze_scale_factor = 0.005,
  autovacuum_vacuum_cost_delay = 2
```
